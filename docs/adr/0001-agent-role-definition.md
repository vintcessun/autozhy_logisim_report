### 一、 内容解析智能体 (Content Parsing Agent)

**文字说明：** 这是整个流水线的“前锋”。它的核心职责是处理非结构化和半结构化的输入数据。

1. **递归解压**：使用 Python 脚本在 `auto_report/data_in/` 中层层剥开提交的压缩包，提取出所有的 `.circ` 文件、`.pdf` 指导书和 `.docx` 报告样本，并将其平铺到工作区。
2. **内容剥离**：利用 `pdfplumber` 和 `python-docx`，精准定位并提取固定的文本。例如，直接从样本中“抠出”**1.实验环境**和**2.实验目的**的纯文本，以及 `.docx` 中单独提出的思考题，存入内存供后续使用。
3. **任务构造**：生成一份结构化的任务清单（JSON 格式），标明哪些文件属于“3.1 验证性实验”，哪些属于“3.2 设计实验”。

### 二、 验证性实验智能体 (Verification Experiment Agent)

**文字说明：** 这是一个“执行者”。当从解析智能体拿到具体的 `.circ` 文件和任务要求后，它开始工作：

1. **任务分发**：为每一个验证条目（如“验证 4 路组相联”）拉起一个专门的**子智能体 (Sub-Agent)**。
2. **精准渲染与截图**：子智能体调用 Logisim 并挂载 `ui-tars` 模型。需要注意的是，因为报告要求“不需要包含电路”，子智能体需要向视觉模型下达精确的指令：*“定位到数码管/探针等输出结果区域，并仅截取该局部区域。”* 3. **状态注入**：通过驱动模拟鼠标拨动开关（例如操作表示控制位的变量 `writeEnableCard` 或 `readEnableCard`），运行到指定时钟周期。
3. **单点分析**：基于局部截图和题目要求，子智能体生成初始的“实验分析”草稿，连同截图路径一并向上层返回。

### 三、 设计性实验智能体 (Design Experiment Agent)

**文字说明：** 这是系统中最复杂、技术含量最高的“创造者”，采用**Actor-Critic (演员-评论家) 闭环架构**：

1. **参考与迁移**：解析智能体提供基础电路，设计智能体首先读取基础 `.circ` 的 XML 结构，理解其拓扑。
2. **代码级生成 (Sub-Agent)**：子智能体（Actor）不通过鼠标去连线（视觉拖拽极易出错），而是直接生成/修改 `.circ` 的 XML 代码。它计算好 `<comp>` 和 `<wire>` 的坐标，输出 XML 补丁，然后计算连线的可达性确保所有线路都准确的连线，先进行一轮小的验证直到通过所有线路的可达性验证（针对每次LLM新添加的所有线路都必须可达）。
3. **闭环修改评测 (Critic)**：设计智能体加载这个生成的 `.circ`。如果不符合要求（如存在悬空连线、或红线报错），将错误日志连同界面的全景截图打回给子智能体要求重写，直到无错闭环。
4. **全景截图与流转**：一旦通过自检，设计智能体进行**全屏截图**（满足 3.2 报告中包含完整电路图的要求）。随后，将这个合法的电路移交给**验证性实验智能体**，执行后续的输入输出测试和结果截图。

### 四、 实验报告智能体 (Report Writing Agent - Orchestrator)

**文字说明：** 它是总指挥和最终的“文字润色师”。它掌握着生成 `.md` 文件的绝对控制权：

1. **全局调度**：控制验证和设计智能体的启动顺序。
2. **上下文精简 (Context-Aware Refinement)**：前置智能体生成的“实验分析”往往为了严谨而显得冗长。报告智能体会结合全局上下文，将文字进行拟人化的修剪，使其读起来像是一份思路连贯、详略得当的优秀学生报告，而不是机器生成的长篇大论。
3. **特定问题插入**：严格按照 `.docx` 提取出的顺序，在对应的实验节点插入关于特定译码器或缓存缺失率的解答。
4. **双重校验 (Double Check)**：每拼接完一个模块（如 3.1 节），报告智能体会进行一次自我反射检查——*“这个部分是否包含 1张结果图 + 1段分析？” “3.2 部分是否包含 1张全图 + 1张结果图 + 1段分析？”* 确认无误后，将占位符替换为相对路径，持久化写入 `auto_report/output/实验报告.md`。
