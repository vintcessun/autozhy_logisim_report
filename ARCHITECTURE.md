# 项目架构设计文档

## 1. 架构总览 (Architecture Overview)

本项目是一个高度模块化的**层次型多智能体系统 (Hierarchical MAS)**，专为传统 GUI 桌面软件（Logisim）的自动化操作与实验分析设计。

系统由四个核心智能体协同工作，深度融合了 **Google 云端大语言模型 (Gemini 生态)** 与 **本地边缘模型 (Gemma 4 / ui-tars)**。通过代码级闭环验证与精准的视觉操控，系统能够实现从“非结构化实验要求”到“标准化实验报告”的端到端全自动生成。

------

## 2. 四大核心智能体工作流 (Agent Workflow)

具体参考`./docs/adr/0001-agent-role-definition.md`

系统的核心业务逻辑严格遵循四大智能体的职责边界：

### 2.1 内容解析智能体 (Content Parsing Agent)

- **职责**：数据摄入与任务拆解。
- **执行逻辑**：
  1. 递归解压 `data_in/` 目录下的嵌套压缩包，提取 `.circ` 文件。
  2. 使用 Python 库（如 `pdfplumber`, `python-docx`）提取实验环境、实验目的以及具体的题目要求。
  3. 将非结构化文本转化为结构化的 Task JSON，分发给下层智能体。

### 2.2 设计性实验智能体 (Design Experiment Agent)

- **职责**：电路拓扑的生成、验证与全景捕获。
- **执行逻辑 (无视觉 Actor-Critic 闭环)**：
  1. **Actor (生成)**：大模型基于题目要求，直接生成/修改 `.circ` XML 代码。
  2. **Critic (校验)**：**不依赖截图**，而是直接将 XML 喂给本地的小模型或 Python 静态脚本，进行图论可达性扫描与语法校验。
  3. **闭环**：若存在悬空连线或语法错误，直接以文本错误日志打回修改，直到逻辑自洽。
  4. **执行与流转**：XML 验证通过后，调用 Logisim 打开文件，使用 `ui-tars` 获取焦点并全屏截图（电路设计图），随后将其移交给验证性实验智能体进行功能测试。

### 2.3 验证性实验智能体 (Verification Experiment Agent)

- **职责**：GUI 交互、状态驱动与结果捕获。
- **执行逻辑 (原生集成视觉与操控)**：
  1. 挂载本地 `ui-tars` 模型，申请全局屏幕互斥锁 (Mutex)。
  2. 强制置顶 Logisim 窗口并修正系统 DPI 缩放。
  3. 根据测试用例，驱动 `PyAutoGUI` 拨动特定引脚，步进时钟（带有单步最多 10 次等灵活的熔断/重试机制）。
  4. 截取最终状态的局部结果图（如数码管/探针），释放屏幕锁。

### 2.4 实验报告智能体 (Report Writing Agent - Orchestrator)

- **职责**：全局调度与文档生成。
- **执行逻辑**：
  1. 作为 Orchestrator，统筹前置智能体的执行顺序。
  2. 接收来自验证/设计智能体的“最终图片”和基础数据。
  3. **杂活下放**：调用轻量级模型（如 `gemini-3-flash` 或本地 `gemma4`）基于最终截图生成对应的段落分析文本。
  4. **排版输出**：将所有文本与图片相对路径整合，严格按照 `docs/examples/实验报告.md` 的示例格式，输出最终的 Markdown 文件至 `output/` 目录。

## 3. 异构模型路由策略 (Heterogeneous Model Router)

系统根据任务特性，将计算任务动态路由至不同的模型节点，以实现成本、速度与精度的最优解：

- **主思考大脑 (gemini-3.1-pro)**
  - 负责高维度的复杂推理，如设计性智能体中的电路架构规划、复杂测试用例的生成、以及 Orchestrator 级的异常决策。
- **高频杂活中枢 (gemini-3-flash / gemma4 本地)**
  - 负责执行具体的文字撰写任务（如基于最终截图撰写实验分析）、XML 格式化清理、以及简单文本的提取转换。
- **原生视觉操控 (ui-tars 1.5 - qwen25vl)**
  - 内嵌于验证与设计智能体中。完全离线运行，专职将 Logisim 屏幕画面转化为 `PyAutoGUI` 的绝对物理执行坐标。

## 4. OS 与物理机交互层 (OS & Host Interaction)

为了保障验证和设计智能体在物理机上调用 GUI 时的绝对稳定，底层封装了以下系统级保护机制：

1. **屏幕资源互斥锁 (Screen Mutex Lock)**
   - 基于 `asyncio.Lock()`，确保多个子任务在排队操控 Logisim 时，同一物理时刻只有一个任务能拥有鼠标和焦点。
2. **强行置顶与防闪烁 (Win32 API Hook)**
   - 注入底层系统事件，破解 Windows 后台防流氓机制，确保截图前 Logisim 窗口绝对处于最上层且已渲染完毕。
3. **物理像素绝对映射 (DPI Awareness)**
   - 强制接管操作系统的 DPI 缩放（`SetProcessDpiAwareness`），确保大模型输出的坐标与鼠标点击的物理像素 1:1 对齐。
4. **防暴走机制 (Fail-Safe)**
   - 开启鼠标角落强制熔断功能，防止大模型幻觉导致的 GUI 误操作。

## 5. 文件系统与排版标准

系统工作区的最终产物实行严格的隔离与路径约束，确保交付物开箱即用：

- **输出目录结构**：

  Plaintext

  ```
  output/
  ├── {实验名称}.md                 
  └── {实验名称}.assets/            
      ├── image-20260410142917.png  
      └── ...
  ```

- **排版约束**：

  最终文档的生成严格锚定 `docs/examples/实验报告.md` 的规范。无论是标题层级、图片占位（使用相对路径如 `./{实验名称}.assets/`），还是思考题的穿插顺序，均需实现对样本格式的像素级/字符级对齐。
