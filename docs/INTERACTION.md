# 开发者交互协议

## 1. 核心方针：主动质疑与知识固化

作为本项目（`autozhy_logisim_report`）的辅助开发 AI，你的首要任务是保持代码架构的严谨性。当你在编写逻辑或处理数据管线发现歧义时，必须遵循以下流程：

**发现歧义 -> 挂起并询问 -> 获取共识 -> 新建文档持久化 -> 继续编码**

------

## 2. 挂起与询问触发点 (When to Stop & Ask)

在 Vibe Coding 过程中，遇到以下情况必须立即停止生成代码并向用户确认：

### 2.1 协议与实现冲突

- **场景**：你发现 `PIPELINE.md` 中定义的组件提取逻辑，在处理 `32位快速加法器.circ` 的实际 XML 结构时无法实现（例如 XML 里没有预期的属性）。
- **动作**：停止编写解析代码。询问用户：“XML 结构与 Pipeline 协议不符，是修改协议还是针对此文件写特殊逻辑？”

### 2.2 命名与风格模糊

- **场景**：需要新建一个处理时钟频率的类，但不确定是否需要像 `EnableCard` 那样加特殊后缀。
- **动作**：询问用户：“对于时钟控制类，是否需要遵循 `ClockControlCard` 这种特定的命名风格？”

### 2.3 业务逻辑断裂

- **场景**：你正在写 `Design Agent` 的 XML 修复逻辑，但你不清楚当两个 Tunnel 重名但位宽不同时应该报错还是自动忽略。
- **动作**：停下来询问。

------

## 3. 知识持久化：docs/context/ 机制

为了防止讨论出的结果在对话刷新后丢失，**每一次重要的询问与结论都必须转化为文档**。

### 3.1 操作规范

当你从用户处获得一个明确的逻辑结论后，请主动提议或直接执行：

1. **新建文件**：在 `docs/context/` 目录下创建 `.md` 文件。
2. **命名参考**：`resolved_{issue_name}.md` 或 `logic_{component_name}.md`。
3. **内容模板**：
   - **Context**: 之前遇到了什么歧义。
   - **Decision**: 用户最终确定的方案。
   - **Code Impact**: 这对后续代码（如 `main.py` 或 `prompts/`）有什么具体影响。

### 3.2 目录示例

执行完成后，你的目录结构应自动演变为：

Plaintext

```
├─docs
│  ├─context           <-- AI 的增量知识库
│  │    resolved_tunnel_naming.md
│  │    logic_adder_carry.md
```

------

## 4. 任务预览与大纲确认 (Outline Gate)

在进行大规模的代码编写（如实现一整个 Agent 类）之前，必须输出一个大纲供用户审核：

1. **输出结构**：例如：“我准备在 `main.py` 中新增 `SimulationManager` 类，包含 `start()` 和 `stop_on_error()` 方法。”
2. **确认**：等待用户回复 `Y` 或修改意见。
3. **记录偏好**：如果用户说“我不喜欢用 try-except，我喜欢用错误码”，请立刻将此记录到 `docs/context/coding_preference.md`。

------

## 5. 编码风格强制约束

在 Vibe Coding 时，请无条件遵守以下规则。若你认为规则不合理，**先提问，禁止私自修改**：

- **变量/函数**：`snake_case`（如 `screen_lock`）。
- **类名**：`PascalCase`（如 `EnableCard`）。
- **路径**：调试信息必须输出到 `./debug/`，最终产物必须输出到 `./output/`。
- **互斥锁**：所有 GUI 操作必须包裹在名为 `screen_lock` 的异步锁内。

------

### 给 AI 助手的 Checklist：

- [ ] 我刚才是否在凭直觉写代码？（如果是，停下来问用户）
- [ ] 讨论出的新规则是否已经写到了 `docs/context/` 里的新文件？
- [ ] 我是否遵循了 `PascalCase` 的类名规范
