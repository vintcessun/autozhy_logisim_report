### 💡 全局前提
- 当前仿真环境已经切换到目标电路，不要输出 `switch_circuit`。
- 只输出 JSON 数组，不要输出解释文字。

### ✅ 强约束（必须遵守）
1. 标签必须使用原文，不允许翻译/改写/臆测。
   - 例如有“复位”就必须写“复位”，不能写“Reset”。
2. `set_value.target` 只能是输入引脚。
3. 读/验类目标 (`get_value`/`check_value`/`tick_until.target`/`run_until_stable_then_tick.target`) 必须来自当前电路真实标签。
4. 所有 JSON 数值字段使用十进制表达（可带负号），且要在电路位宽允许范围内。
5. 若任务需要“等待稳定后再多跑若干周期”，优先使用 `run_until_stable_then_tick`，不要手工拼大量 tick。

### 🛠 支持动作
1. `set_value`
   - 格式: {{ "action": "set_value", "target": "输入引脚名", "value": "十进制值" }}
   - 作用: 给输入引脚/可写组件赋值。

2. `get_value`
   - 格式: {{ "action": "get_value", "target": "引脚/组件名" }}
   - 作用: 读取当前值。

3. `tick_until`
   - 格式: {{ "action": "tick_until", "target": "观测点", "expected": "期望值", "max": 100, "clock": "可选输入时钟引脚" }}
   - 规则:
     - `max` 至少为 100。
     - `clock` 若提供，必须是输入引脚；API 会每周期自动脉冲该引脚。

4. `check_value`
   - 格式: {{ "action": "check_value", "target": "引脚/组件名", "expected": "期望值" }}
   - 作用: 断言值是否符合预期。

5. `load_memory`
   - 格式: {{ "action": "load_memory", "target": "ROM标签", "txt_path": "txt绝对路径" }}
   - 参数来源:
     - `target`: 当前电路中真实 ROM 或者 RAM 标签（来自任务原文或给定引脚/组件信息）。
     - `txt_path`: 任务资料中提供的真实文件路径，不可编造。
   - 文件格式:
     - 第一行必须是 `v2.0 raw`
     - 后续是十六进制数据流，按地址 0 顺序写入
     - 超 ROM 容量会失败

6. `run_until_stable_then_tick`
   - 格式: {{ "action": "run_until_stable_then_tick", "target": "观测点", "timeout_second": 10, "k": 100, "stable_samples": 5, "poll_ms": 20 }}
   - 参数来源:
     - `target`: 需要判断稳定性的真实观测点标签（优先从输出引脚或任务明确观测点中选）。
     - `timeout_second`: 正数，按任务复杂度给出合理值（建议 10 起）。
     - `k`(可选): 稳定后额外 tick 次数；无额外要求可设 0。
     - `stable_samples`(可选): 连续相同样本数，默认 5。
     - `poll_ms`(可选): 轮询间隔毫秒，默认 20。
   - 语义顺序:
     1) Reset
     2) 以 4.1kHz 连续时钟运行
     3) 轮询 `target` 至稳定
     4) 额外执行 `k` 次 tick
   - 超时语义:
     - 若在 `timeout_second` 内未稳定（或额外 tick 阶段超时），视为失败；参数不要设置过小。

7. `run_tick`
   - 格式: {{ "action": "run_tick", "tick_count": 2, "req_id": "tick-1" }}
   - 作用: 在冻结状态下执行指定数量的仿真 tick。每个 tick 使内部 `Clock` 组件推进一步。
   - 参数:
     - `tick_count` (必需): 执行的 tick 次数，必须大于 0。
       - 示例：`2` = 1 个完整时钟周期 (CLK: 0→1→0)
       - 示例：`4` = 2 个完整时钟周期
     - `req_id` (可选): 请求标识符，用于追踪请求。
   - 响应:
     ```json
     {{ "status": "ok", "req_id": "tick-1", "ticks": 2 }}
     ```
   - 使用场景: 用于简单的单步仿真，无需 `run_until_stable_then_tick` 的复杂性。适用于：
     - 手动逐步遍历逻辑
     - 在截屏前运行已知数量的时钟周期
     - 在 `load_memory` 或 `set_value` 后验证状态转换

### ⚠️ 动作编排规则
1. 初始化优先: 在任何 `tick_until`/`run_until_stable_then_tick` 之前，先用 `set_value` 设置必要输入（复位、使能、模式位等）。
2. 时序任务: 涉及必须使用时钟才能得出正确结果时，优先使用 `tick_until` 的 `clock` 或宏命令，不要手工高低电平拼接多步。
3. 结果校验: 对目标要求涉及的关键输出，使用 `check_value` 明确断言；必要时再补 `get_value` 读取。
4. 最小充分: 动作序列应尽量短，但必须完整覆盖目标验证点。
